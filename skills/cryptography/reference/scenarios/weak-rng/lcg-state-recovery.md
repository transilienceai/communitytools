# LCG State Recovery

## When this applies

- Random values are generated by a Linear Congruential Generator: `state = (a * state + c) mod m` with output either the full state or a high-bits truncation.
- You can observe several consecutive outputs (or outputs at known offsets).
- Trigger keywords: "LCG", "linear congruential", `rand()`, glibc `random()`, Java `Random`, MS Visual C++ `rand()`.
- Constants `(a, c, m)` may be known (standard PRNGs) or unknown (custom LCG).

## Technique

Three regimes:

1. **Full-state output, all params known.** One sample reveals the seed; use the recurrence directly.
2. **Full-state output, params unknown.** Recover `m`, then `a`, then `c` from a handful of samples using gcd.
3. **Truncated output (top/bottom `k` bits hidden).** Lattice attack — reduce to a hidden-number problem.

## Steps

### Regime 1 — Full state, params known

```python
# state_n+1 = (a * state_n + c) % m known
# next = (a * observed + c) % m
predicted = (a * observed + c) % m
```

Roll forward / backward as needed.

### Regime 2 — Recover `(a, c, m)` from outputs

Given `s0, s1, s2, s3, s4, s5`:

```python
# Step 1: recover m
diffs = [s[i+1] - s[i] for i in range(5)]
T = [diffs[i+2]*diffs[i] - diffs[i+1]**2 for i in range(3)]
from math import gcd
m_candidate = abs(gcd(*T))   # take gcd of all Ts
# remove small factors (divide out common 2/3/5 factors carefully)

# Step 2: recover a
a = ((s2 - s1) * pow(s1 - s0, -1, m_candidate)) % m_candidate

# Step 3: recover c
c = (s1 - a * s0) % m_candidate
```

Verify: `s2 == (a*s1 + c) % m`. Adjust `m_candidate` (try dividing by small primes) if not.

### Regime 3 — Truncated outputs (lattice)

If only top `k` bits of state are output: this is the **truncated linear congruential** problem. Build a lattice from the recurrence and the bit truncation, run LLL. See `scenarios/lattice/lll-basis-reduction.md` for setup; the hidden-number-problem template applies directly.

For glibc `random()` (additive feedback, not strict LCG), use `rngtools` (Python). For Java `java.util.Random` (48-bit LCG outputting top 32 bits per call), 2 outputs are enough; brute-force the 16 missing bits in `2^16` time.

```python
# Java Random: state = (a * state + c) & ((1<<48) - 1), output = state >> 16
# Given two outputs r0, r1:
a, c, mask = 0x5DEECE66D, 0xB, (1<<48) - 1
for low in range(1<<16):
    s0 = (r0 << 16) | low
    s1 = (a*s0 + c) & mask
    if (s1 >> 16) == r1:
        seed = s0
        break
```

## Verifying success

- Predict the next output; submit and confirm.
- Roll back to seed; verify earlier outputs match.
- For RNG used as key generator: derive the key and decrypt a known ciphertext.

## Common pitfalls

- **Mistaking glibc `random()` for an LCG.** It uses an additive feedback shift register; standard LCG attacks don't apply directly.
- **Output transformation.** Many APIs output `state mod range`, losing bits; account for the modulus when fitting samples.
- **Negative `c`.** Some textbooks define LCG with `c=0` (multiplicative). Confirm by attempting recovery with both.
- **Modulus has small factors.** If `m = 2^k`, things behave differently from prime `m` (use `pow(x, -1, m)` only on odd residues).
- **Java's seed scrambling.** `new Random(seed)` XORs with `0x5DEECE66D`; account for this when tying seed to source value.

## Tools

- Pure Python; gcd + modular inverse is enough for full-state recovery.
- `randcrack` (Python) for MT19937 (see neighbor file).
- `rngtools` for glibc / BSD `random()` reverse engineering.
- For truncated LCG: SageMath or fpylll for the lattice reduction.
